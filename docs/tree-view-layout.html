Tree view — current layout method

How the hourglass chart in src/client/components/tree-view/ is actually built, written in the vocabulary defined by tree-view.html and the decisions in ADR 0001.

The Sub-layout calculus

Every recursive unit — AncestorTree, DescendantTree, Couple, Sibship, FocusRow — returns the same shape, defined in helpers.ts:36: 

interface SubLayout {
  leftWidth:  number;            // extent left  of the pivot
  rightWidth: number;            // extent right of the pivot
  nodes:      PositionedPerson[];
  lines:      Line[];
}

The unit's Pivot always sits at local x = 0. Composers stitch units together by calling shiftLayout(sub, dx) to land each pivot at a target X, then unionLayouts to merge them. This contract is load-bearing — switching shape (e.g. to a cumulative cursor or a constraint solver) would mean rewriting every recursive unit (see ADR §"Why this is hard to reverse").

Top-level assembly

buildChart in layout.ts:214 composes the whole chart around Focus at SVG (0, 0):

  1. computeBloodlineParentTrees recurses up paternal and maternal sides via layoutAncestorBranch (layout.ts:46), bottoming out at the levels cap (the Generation limit).
  2. layoutFocusRow lays out everything at Y = 0 — Focus + siblings + Focus's spouses + half-sibships beside them — and reports back parentAnchorX (where the bloodline parent Couple should center above) and stepFams (step-parent slot positions for tie drawing).
  3. composeBloodlineParents places Father/Mother at Y = −ROW_H around that anchorX using the pre-computed sep; stepTies (layout.ts:271) then draws the Tie line from each step-parent box (already shifted in focus-row's output) back to the appropriate bloodline parent, at a slightly offset Y so it stays visually distinguishable from the bloodline Couple Tie when the two cross.

Ancestor recursion

layoutAncestorBranch (layout.ts:46):

  • Base case. Depth at the Generation limit, or no parent family known → return a single bareBox at Y = −depth · ROW_H.
  • Recursive case. Lay out father and mother sub-trees at depth + 1, compute coupleSeparation = max(COUPLE_PITCH, fatherSub.rightWidth + motherSub.leftWidth + COUPLE_GAP) — the deep-branch widths push the Couple apart so sub-trees can't collide; COUPLE_PITCH is the floor for shallow trees.
  • Compose via composeCouple (layout.ts:107), which adds the horizontal Tie between inner box edges and returns an anchorY — the Child anchor from the doc. When both parents are present anchorY = y (Tie midpoint); when only one, it drops to the lone box bottom.
  • A pdrop line from anchorY down to the bloodline child's box top finishes the unit.

Two coupling modes

The same Couple shape, attached two different ways:

  • centeredCoupleSides (layout.ts:135) places father at −sep/2 and mother at +sep/2 around local pivot 0. Used inside the ancestor recursion — the caller will shift the whole unit afterward, so a centered local frame is what each recursive level wants.
  • anchoredCoupleSides (layout.ts:158) places father at anchorX − sep/2 and mother at anchorX + sep/2. Used once, at the bloodline-parent row, so the Couple's Tie midpoint sits directly above the focus-row sibship's bar midpoint.

Sibship orientation rule (asymmetric on purpose)

Focus row — full-sibship birth order, with the Primary spouse inserted adjacent to Focus per the Husband-left convention. The sibship bar visually passes behind any inserted spouse box (boxes render above the bar by z-order).

Ancestor rows — bloodline person at the inner edge of their sibship: Father at the right edge of his sibship (spouse-facing), Mother at the left edge of hers, with aunts/uncles fanning outward in birth order. Strict birth order across the whole row is not preserved here.

The asymmetry is deliberate. Applying Focus's rule recursively at ancestor rows produces visually unworkable layouts where the two sibship bars overlap and the central Couple gets sandwiched between non-bloodline siblings. The inner-edge rule keeps each sibship visually distinct and matches the canonical diagram in tree-view.html. See ADR §"Sibship orientation rule".

Half-sibships

Half-sibling families are not placed by layout.ts. Focus-row owns Y = 0 and assigns each non-mainFam (mother's other husbands' kids, father's other wives' kids) its own packed slot. composeBloodlineParents reads those slot positions back via stepFams and only draws the Tie line from each step-parent box to the appropriate bloodline parent at the parent row — the non-Primary Child anchor rule from the doc.

Why charts are wide

Each coupleSeparation propagates upward via the sub-layout's rightWidth / leftWidth, so a deep branch on one side widens every ancestor Couple above it. Combined with fanning aunts/uncles, multi-spouse Primary-then-fan, and half-sib sibships beside the Focus row, this can produce layouts that feel wider than minimal. The chart accepts width in exchange for never having lines cross boxes uncontrolled — that trade-off is the load-bearing reason the Sub-layout calculus exists (ADR §"Consequences").
