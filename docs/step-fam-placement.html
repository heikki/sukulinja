Step-fam placement — decoupled sep + chronological side

Maintain the block-tree structure from ancestor-refactor.html: step-fams stay as entries in PersonBlock.marriages on Fa / Mo, with the bloodline parent external inside each step-fam FB. Two refinements pin down the layout so it (a) keeps Fa–Mo close together, and (b) places step-fams left/right of the bloodline kid row according to chronological offset — matching pre-refactor behaviour.

Two fixes: (1) Decouple sep widening from step-fam extent. Fa's step-fams must not push Mo outward — Fa–Mo separation should depend only on the childhoodFamily subtrees, exactly as the bloodline couple sep did pre-refactor. (2) Side of the step-fam follows chronology. A step-fam with chronological offset < 0 (before bloodline) lands left of the bloodline kid row; offset > 0 lands right. Same rule as the legacy focus-row pack.

1. The coupling that has to go

When Fa.PB.rightWidth includes the step-fam FB extent, the parent FB's sep widens with Xstep. Fa shifts left at exactly the rate the Tie midpoint moves right relative to Fa — so the half-sib bar's chart-X is locked, independent of Xstep:

half-sib chart X = Fa.chart + Xstep/2 Fa.chart = parent-FB-offset.x − sep/2 sep = Xstep + (Mo.left + 120) // step-fam dominates Fa.right ⇒ half-sib chart X = parent-FB-offset.x − (Mo.left + 120)/2 // Xstep cancels

And while Xstep doesn't help half-sib placement, it does blow up Fa–Mo separation (sep grows linearly with Xstep). Both problems disappear once Fa.PB's step-fams no longer feed sep widening.

2. Decouple: PersonBlock exposes couple-width and activeMarriageIndex

Two related changes to PersonBlock:

  1. Split the width fields. coupleLeftWidth / coupleRightWidth come from own box + childhoodFamily only — used by the parent FB for sep widening, so Fa–Mo separation depends only on the bloodline ancestor chains. The regular leftWidth / rightWidth include marriages, used for outer packing and the overall chart bounding box.
  2. Marriages are one chronological list with an activeMarriageIndex marking the active marriage — the one rendered in primary-couple style. Slots before/after the active index are step-fams (non-primary). A null entry at the active slot means "active marriage is owned by an outer Block" — that's exactly the case for bloodline ancestors, whose bloodline marriage is drawn by the parent FB above.
class PersonBlock extends Block { readonly personId: number; readonly childhoodFamily: FamilyBlock | null; readonly marriages: readonly (FamilyBlock | null)[]; // chronological readonly activeMarriageIndex: number | null; // index in marriages[] // Own box + childhoodFamily extent only — drives Fa–Mo sep. readonly coupleLeftWidth, coupleRightWidth; // Own box + childhoodFamily + marriages — drives outer packing. readonly leftWidth, rightWidth; } // Roles → shape: focus PB: marriages = [primary, secondary, ...], activeMarriageIndex = 0 sibling PB: marriages = [primary], activeMarriageIndex = 0 ancestor PB: marriages = [...stepFams interleaved with `null` at the bloodline slot...], activeMarriageIndex = (index of the `null` slot) bare PB: marriages = [], activeMarriageIndex = null // Parent FB: sep = max(COUPLE_PITCH, Fa.coupleRightWidth + Mo.coupleLeftWidth + COUPLE_GAP) parent-FB.leftWidth = max over children of (child.x − child.leftWidth) parent-FB.rightWidth = max over children of (child.x + child.rightWidth)

3. Step-fam side follows chronology (pre-refactor rule)

Chronological order is the order of the bloodline parent's spouseFams in the data model. Each step-fam's signed offset from the bloodline marriage is just its position in PB.marriages minus activeMarriageIndex. Negative offset → place on the bloodline kids' left; positive → right. Within each side, sort by |offset|; the closest-to-bloodline entry sits adjacent to the bloodline kid row, others fan further out.

buildAncestorPB(parent, bloodlineFamId, ix): fams = ix.spouseFamsByPerson[parent] // already chronological bloodIdx = fams.indexOf(bloodlineFamId) entries = fams.map((fam, i) => fam.id === bloodlineFamId ? null : buildStepFamFB(fam, ix)) return new PersonBlock({ childhoodFamily: buildChildhoodFamily(parent, depth=1, ix), marriages: entries, activeMarriageIndex: bloodIdx, }) // At layout time, for each non-null marriages[i] on Fa.PB or Mo.PB: offset = i − Fa.activeMarriageIndex if offset < 0: place this step-fam left of bloodline kid row if offset > 0: place this step-fam right of bloodline kid row return out leftStepFams = out.filter(s => s.offset < 0).sort by offset asc // innermost first rightStepFams = out.filter(s => s.offset > 0).sort by offset asc // innermost first

4. Sizing the step-spouse offset

Now that sep is decoupled, Fa.chart is known after Pass 1 (the bloodline pack) — Xstep is a free parameter inside the step-fam FB. Use the non-primary anchor convention from CONTEXT.md (half-sib bar drops from the step-parent's box bottom) so the bar moves with Xstep linearly.

// Pass 1 — bloodline kid pack (independent of Fa.PB / Mo.PB step-fams) pack = packBlocks(bloodline kid PBs) parentFBOffsetX = −(focus.parent_local.x) bloodlineLeft = parentFBOffsetX + (pack.minX − pack.barMid) bloodlineRight = parentFBOffsetX + (pack.maxX − pack.barMid) // Pass 2 — build Fa.PB / Mo.PB (only childhoodFamily here; no step-fams yet) FaPBPre, MoPBPre = buildAncestorPB(..., includeStepFams=false) sep = max(COUPLE_PITCH, FaPBPre.coupleRight + MoPBPre.coupleLeft + COUPLE_GAP) FaChartX = parentFBOffsetX − sep/2 MoChartX = parentFBOffsetX + sep/2 // Pass 3 — size each step-fam's Xstep from the bloodline edge on its side let cursorLeft = bloodlineLeft − SIBLING_GAP // right edge of next "before" step-fam ≤ cursorLeft let cursorRight = bloodlineRight + SIBLING_GAP // left edge of next "after" step-fam ≥ cursorRight for sf in rightStepFams (innermost first): parentChartX = sf.parent == Fa ? FaChartX : MoChartX fanDir = husband-left says step-spouse on Fa's right / Mo's left Xstep = (cursorRight − parentChartX) + sf.barHalfLeft + BOX_W/2 cursorRight = parentChartX + Xstep + sf.barHalfRight + BOX_W/2 + SIBLING_GAP for sf in leftStepFams (innermost first): parentChartX = sf.parent == Fa ? FaChartX : MoChartX Xstep = (cursorLeft − parentChartX) − sf.barHalfRight − BOX_W/2 // negative when step-fam is left of parent cursorLeft = parentChartX + Xstep − sf.barHalfLeft − BOX_W/2 − SIBLING_GAP

Fa(9)–Mo(10) separation is back to the bloodline-only value (sep = 424, the same as pre-refactor for this case). The step-wife (77, chronological offset +1) lands right of bloodline 17 — the rightmost bloodline kid — and its half-sib bar drops from her box bottom.

bloodline (Fa, Mo, focus, siblings) focus step-fam (inside Fa.marriages)

Fa–Mo are at the tight pre-refactor separation. The step Tie (long horizontal at Y = parent row − 6) crosses Mo's column; this is the same behaviour pre-refactor had for "after" step-fams, drawn slightly above the bloodline Tie so the two don't overlap.

6. Chronological-before step-fam (mirror case)

For a step-fam with offset < 0 (the bloodline parent's marriage to step-spouse predates the bloodline marriage), the algorithm in §4 produces a negative Xstep — the step-spouse lands left of Fa, beyond the bloodline kid row's left edge. The step-fam FB's husband-left convention is flipped for these cases (step-spouse on Fa's left while Fa is still the husband). This matches what pre-refactor did: those step-fam slots packed left of the bloodline siblings, and the step Tie crossed leftward.

Husband-left convention violation acknowledged: when the step-fam sits left of the bloodline couple but Fa is still the husband (M), the step Tie reverses (wife on left of husband). Pre-refactor had the same visual layout but drew the step-spouse as a lone-parent block + cross-block Tie, side-stepping the question. We're choosing to keep the FB intact (Fa external + step-spouse internal, Tie inside) and accept the convention flip.

7. Trade-offs

Property Effect
Block-tree shape Unchanged. Step-fams stay inside PersonBlock.marriages; FB is still "one family"; no new block type; no cross-block lines.
Fa–Mo separation Matches pre-refactor — determined only by childhoodFamily subtrees, not by step-fams (decoupling in §2).
Step-fam side Matches pre-refactor — chronological offset relative to bloodline fam (§3).
Step Tie length Long. It crosses the bloodline parent on the opposite side (Mo's column for right step-fams, Fa's column for left step-fams). Mitigated visually by the Y offset (STEP_TIE_Y_OFFSET) above/below the bloodline Tie. Pre-refactor had the same property.
Husband-left convention Held within the FB for offset > 0 step-fams. Flipped for offset < 0 step-fams (step-spouse appears on the "wrong" side of the bloodline husband). Pre-refactor avoided this by drawing step-spouse as a lone block + cross-block Tie; we keep the FB and accept the flip.
Build order Two passes. Pass 1 packs bloodline kids; pass 2 builds Fa.PB / Mo.PB with Xstep sized from the bloodline row's chart extents.
Half-aunts/uncles (depth ≥ 2) Still deferred. Would require the same two-pass coordination one level deeper.
